D — Key Technical Decisions

Decision 1

Self-hosted Langfuse vs Datadog LLM Observability

✗  Datadog LLM Obs (SaaS) — rejected
✓  Self-hosted Langfuse — chosen
Financial transaction data cannot leave the GCP project boundary. Self-hosted Langfuse means zero PII escapes the network perimeter.
Decision 2

pgvector vs Vertex AI Matching Engine

✗  Vertex AI Matching Engine — rejected
✓  pgvector running co-located with agents — chosen
Cost and latency. Local vector store with no API round-trip for every RAG query. Matching Engine adds per-query cost and network RTT.
Decision 3

GKE Autopilot vs fragmented GKE + Cloud Run

✗  Separate GKE node pool + Cloud Run services — rejected
✓  Unified GKE Autopilot — chosen
Consolidated serverless footprint, simpler scaling, single Terraform scope. Fragmented infra adds operational overhead without benefit at this scale.
Decision 4

MemorySaver vs Redis (local development)

✗  Redis for all environments — rejected
✓  MemorySaver local dev, Redis production — chosen
Redis service container has compatibility issues during local development. MemorySaver provides identical LangGraph checkpointing semantics without the container dependency.

Reward Design

Hybrid reward combines objective SLO metrics with qualitative LLM-as-Judge evaluation.

  • Objective component: SLO attainment — did MTTA < 90s? did MTTR < 8min?
  • Qualitative component: LLM-as-Judge (gpt-4o-mini) scores the action on 3 dimensions — correctness, speed, and business impact
  • Why hybrid? SLO metrics alone are sparse (only resolved at incident end). LLM judge provides dense intermediate signal.
  • A/B comparison: cumulative reward graphed live in Next.js dashboard

Adversarial Hardening

  • AST-validated SQL generation — DDL injection (DROP, DELETE, ALTER) blocked at parse level before execution
  • Hard-capped result sizes — prevents bulk data exfiltration through the agent interface
  • LangFuse audit trail — every AI-generated query and its result logged with session scoping and full traceability

Responsible AI Architecture

  • Agent autonomously generates and executes read-only SQL — but DDL/DML (DROP, DELETE, ALTER) is rejected at the AST layer before execution, so it can query but never mutate the database
  • Gradio UI with authentication — no unauthenticated access to the agent interface
  • Persistent cross-session memory via PostgresSaver — agent maintains context across conversations
  • Production deployment: Docker Compose (dev) → Vertex AI Reasoning Engine / GKE Autopilot

3 Constrained LLM Generation

CLV Predictor — DSPy Integration

Same project as Tracks 1 and 2 — presenting only the DSPy component. This is not prompt engineering. This is an architectural constraint that makes LLM outputs structurally reliable. The reliability comes from the system design, not from better prompts.

Standard LLM approach

Prompt the model with instructions and hope the output stays grounded. The model can and will produce generic, hallucinated content that sounds reasonable but is not anchored to the specific account's data.

DSPy ChainOfThought approach

dspy.ChainOfThought via OutreachSignature forces the model to reason through deterministic ML inputs (CLV Tier, Churn Risk, Recommendation Space) before writing. Every output is anchored to specific ML predictions for a specific account.

Technical Implementation

  • DSPy ChainOfThought embedded directly in the FastAPI inference service — unified container
  • Single POST /generate/outreach endpoint — zero-latency access to pre-loaded feature matrices
  • Azure OpenAI gpt-4o endpoint, max_tokens=2048
  • Joblib scikit-learn models and DSPy logic in the same API container — no cross-service latency
