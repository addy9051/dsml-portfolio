Data Infrastructure

Snowflake + dbt Star Schema

  • RAW → STAGING → MARTS layered warehouse with dbt 1.9 transformation layer
  • dim_accounts — zero-null hardened dimension with synthesized ACV metrics
  • fact_bookings — granular transaction layer for compliance & leakage detection
  • fact_ml_churn — survival analysis output with support ticket counts
  • fact_ml_cross_sell — recommendation matrix unpivoted for categorical analysis
  • COALESCE filters across all models — Power BI never displays (Blank) or Null

Airflow DAG Orchestration

  • data_ingestion DAG — pulls raw data into Snowflake RAW layer
  • snowflake_loader DAG — triggers dbt transformation runs through staging to marts
  • Databricks job triggered for PySpark feature engineering and model retraining
  • Pipeline is fully orchestrated — no manual intervention required between raw data and serving

Model Governance — MLflow Registry

Four production models tracked, versioned, and served from a single MLflow registry. Bayesian hyperparameter optimisation (Optuna) on all models. Walk-forward cross-validation (not standard k-fold) to prevent temporal leakage on time-indexed account data.

FMCG Retail Lakehouse ETL Data Engineering Project

Post-merger data consolidation problem: two separate FMCG company datasets, incompatible schemas, no shared data contract. Solution: Medallion Architecture on Databricks — each layer has a defined contract and a defined purpose. If a transformation is wrong, replay from Bronze — no re-ingestion needed.

Engineering Decisions

  • Bronze = untransformed truth. Schema-on-read, zero modifications, full audit trail. Every Silver/Gold error traces back here.
  • PySpark for Silver — distributed processing handles schema harmonisation across two incompatible company datasets at scale
  • Incremental load pattern on Gold fact tables — processes only delta records, not full refreshes
  • Unity Catalog — access governance and metadata management across both company datasets

Analytics Use Cases Enabled

  • Sales performance — revenue trends by product, region, and time
  • Inventory analysis — stock levels and turnover rates
  • Customer segmentation — buying behaviour across both companies
  • Post-merger comparison — side-by-side metrics before and after consolidation
  • Databricks Genie — natural language querying of Gold layer, eliminating analyst dependency for routine questions

3 Production ML Serving

Player Churn Prediction & Engagement Optimisation Platform

The engineering story here is a training-serving skew fix and a PySpark migration — not the model accuracy. Distributed feature engineering on S3/EMR datasets, a systematic fix for scale divergence at inference, and a secured FastAPI service with async prediction logging for downstream drift monitoring.

The Engineering Story

Training-Serving Skew Fix

Problem: Z-score scaling applied separately in Spark during training and independently in the API at inference caused distribution mismatch — the model saw different feature distributions at serving time.

Fix: Integrated StandardScaler directly into a sklearn Pipeline object. The fitted scaler serialised during training is automatically applied at inference — zero divergence guaranteed.

PySpark Migration

  • Transitioned from Pandas to PySpark for distributed feature engineering over scale-out S3/EMR datasets
  • Spark StringIndexer mappings exported to category_mappings.pkl for accurate categorical encoding at inference
  • Walk-forward chronological feature construction — no look-ahead leakage

Serving Infrastructure

  • FastAPI POST /predict with X-API-Key authentication
  • MODEL_PATH + MAPPINGS_PATH env vars — S3/GCS compatible at startup
  • Timestamped joblib artifacts + dynamic advanced_xgb_latest.pkl alias
  • Async prediction logging → prediction_logs.csv for drift monitoring and lift analysis
