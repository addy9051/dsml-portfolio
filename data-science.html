1 Lead Project

Customer Lifetime Value & Cross-Sell Predictor

Same project as Track 1 — entirely different framing. This page covers the modelling decisions: why each algorithm was chosen, what it assumes, and how evaluation was done honestly. The pipeline story is on Track 1.

Multi-Model ML Platform — Survival Analysis, Density Clustering, and Multi-Label Propensity

Decision 1 — Cox PH for Churn (not binary classification)

The Modelling Decision

Corporate travel churn is right-censored data. Some accounts are still active at observation time — they haven't churned yet, but we don't know if or when they will.

A binary classifier answers: did this account churn?
Cox Proportional Hazards answers: when will this account churn, and what is their hazard trajectory over the next N months?

That difference enables proactive intervention months before the churn event — not a post-hoc label.

Evaluation — C-Index = 0.89

Standard AUC does not apply to survival models. C-Index (Concordance Index) is the correct metric: it measures what percentage of account pairs the model correctly ranks by relative churn hazard.

C-Index = 0.89 means the model correctly ranks 89% of account pairs by relative churn risk. Random chance is 0.50. Perfect ranking is 1.00.

Walk-forward cross-validation enforced — no temporal leakage between training and validation windows.

Decision 2 — HDBSCAN + UMAP for Segmentation (not K-Means)

Why Not K-Means

  • K must be pre-specified. K-Means forces you to decide the number of segments before seeing the data. The number of natural tiers in account behaviour is an empirical question, not a modelling assumption.
  • K-Means never abstains. Every account gets forced into the nearest centroid, even genuine outliers. Noise is indistinguishable from cluster members.
  • Centroid geometry. K-Means assumes spherical, equal-variance clusters. Account behaviour does not satisfy this.

Why HDBSCAN + UMAP

  • HDBSCAN discovers K from data density. Three tiers (At-Risk, Core, Growth) emerged organically — not from a pre-set parameter.
  • Noise isolation. Outlier accounts are labelled as noise and excluded — they don't corrupt segment profiles.
  • UMAP before clustering. Preserves non-linear manifold structure better than PCA, which only captures linear variance. UMAP then feeds into HDBSCAN's density estimation.
  • Result: Silhouette = 0.77

Decision 3 — Multi-Output XGBoost for Cross-Sell (not 5 separate models)

The Alternative Rejected

Five separate binary classifiers — one per product (Neo, Expense, Consulting, etc.). Each trained independently, each with its own feature matrix, each requiring its own inference call.

Downsides: 5× API latency, 5× model artifacts to govern, and no cross-product correlation signal during training.

The Decision Made

Single multi-label XGBoost model outputs independent propensity scores for all products in a single inference call.

  • Single model artifact to version and govern in MLflow
  • Implicit learning of inter-product correlations during training
  • Single POST to /predict/cross-sell returns all product scores
  • AUC = 0.92+ across product set with calibrated propensity scores

Decision 4 — LightGBM for CLV Regression (not linear regression or deep learning)

Tabular transaction data is noisy, has missing values, and has complex non-linear feature interactions that linear models cannot capture. Deep learning requires far more data and interpretability becomes a problem.

Native handling

LightGBM handles missing values and noisy features natively — no imputation preprocessing step required.

SHAP explainability

SHAP feature importance explains why each account has a specific CLV — essential for account manager trust.

Results

R² = 0.61, MAE = $14.8K on 42 engineered features. Bayesian hyperparameter optimisation (Optuna). Walk-forward CV.

Player Churn Prediction — Experimentation Angle

Same project as Track 1 — different framing. This page covers the experimentation decisions: problem reframing, custom evaluation metric design, and A/B test structure. The serving infrastructure story is on Track 1.

Experimentation Design — Problem Reframing, Revenue-Weighted Evaluation, ROI-Gated Targeting

Problem Reframing — Changing the Target Variable

Standard definition (rejected)

Churn = player has already left. Label assigned retrospectively. A model trained on this definition learns to identify who left — useful for post-hoc analysis, not for intervention.

Reframed definition (adopted)

Churn = DaysSinceLastLogin > 14. A predictive leading indicator. The model learns to identify players heading toward disengagement while there is still time to intervene.

This is an analytical insight — changing the definition of the target variable changes what the model optimises for. Same data, same algorithm, different prediction horizon and very different product utility.

Revenue-Weighted AUC — Aligning the Loss Function to Business Cost

Standard AUC

Penalises all false negatives equally. Missing a low-value player and missing a high-spend "whale" player have the same cost in the objective function. This is wrong.

Revenue-Weighted AUC

False negatives on high-InGamePurchases players are penalised more heavily. The model's objective function matches the actual business cost function — missing a whale costs more than missing a minnow.

Implemented in evaluate_model.py as a custom sample-weight scheme on the ROC computation.

A/B Experimentation Design

Expected Value Targeting

Intervention decision rule:
(Churn_Prob × LTV)
− Campaign_Cost
> ROI_Threshold Concentrates retention spend on profitable Whales rather than broadcasting uniformly.

ROI-Gated Cohort Selection

  • The Expected Value rule is wired into the A/B testing clustering logic in code — it decides which players enter an intervention cohort
  • Spend is gated to players where predicted ROI clears the threshold, so the treatment population is selected by profitability, not broadcast
  • The framework is built to support a controlled lift test; a completed power analysis with significance results is future work, not a claimed result

Telemetry & Drift

  • Async prediction logging captures player ID, churn probability, LTV score, and timestamp
  • Enables downstream lift analysis on the intervention cohort
  • Enables detection of model performance degradation over time (covariate drift)

Advanced Cardiac Segmentation for Clinical Echocardiography

Comparative evaluation of 3 deep learning architectures for automated Left Ventricle and Left Atrium segmentation in 2D echocardiograms. The core finding: loss function engineering provided a higher performance gain than architectural complexity for this segmentation task.

Architecture Comparison

Baseline — U-Net Cross-Entropy 0.8453 0.9112 0.0979

Key Finding — Loss Function Engineering > Architectural Complexity

Moving from U-Net to Attention U-Net (adding spatial attention gates) improved mIoU by only 0.0018. Moving from Cross-Entropy to Dice + Focal Loss improved mIoU by 0.0056 — roughly 3× the gain of the architectural upgrade.

The reason: the dataset has severe class imbalance between background pixels and thin structures like the myocardial wall. Standard Cross-Entropy treats every pixel equally. Dice + Focal Loss addresses this directly.

Why Dice + Focal Loss

Dice Loss

Measures overlap between predicted mask and ground truth mask. Directly penalises under-segmentation of small structures (the thin myocardial wall) where pixel count is low but clinical importance is high.

Focal Loss

Down-weights easy negatives (background pixels) so the model focuses training signal on hard positives (thin wall pixels). Modulating factor (1-pₜ)² ensures the loss is concentrated where the model is currently wrong.

Data Split — Patient-Level (not Record-Level)

80/10/10 patient-level split (400 / 50 / 50 patients → 1600 / 200 / 200 image-mask pairs). A random record-level split would allow the same patient's images to appear in both training and test sets, inflating evaluation metrics through data leakage. Patient-level splitting prevents this entirely.

Dataset: CAMUS (shoybhasan/camus-human-heart-data) — 6,000 NIfTI image-mask pairs across 500 patients. 2CH and 4CH echocardiographic views, ED and ES frames. 4 segmentation classes: background, LV endocardium, LV myocardium, Left Atrium.
